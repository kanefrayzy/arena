────────────────────────────────────────────────────────────────────────────────
1. КРАТКОЕ ОПИСАНИЕ ПРОДУКТА
────────────────────────────────────────────────────────────────────────────────

Браузерная PWA-игра в жанре real-time 2D top-down арена-шутер 1v1
(в духе Brawl Stars / .io-шутеров).

Три режима матчей:
  FREE    — без денег, для тренировки.
  CASUAL  — фиксированный приз/штраф (по умолчанию ±0.03 USD), включается
            админом, может быть отключён.
  STAKE   — комнаты с фиксированными ставками (1, 5, 10 USD и далее настраиваются
            в админке). Победитель получает 80% банка, 20% — комиссия.

Юридическая модель: skill-game (победа определяется навыком игрока,
никакого RNG в исходе боя).

Валюта: USD.
Платежи: Betra H2H (депозиты + выплаты).

────────────────────────────────────────────────────────────────────────────────
2. КЛЮЧЕВЫЕ ТРЕБОВАНИЯ
────────────────────────────────────────────────────────────────────────────────

Геймплей:
  - 2D top-down арена 1v1.
  - Персонажи с разными статами (HP, скорость, урон, оружие, способность).
  - Скины (могут быть чисто косметическими или модифицировать статы — галка в админке).
  - Server-authoritative: клиент шлёт input, сервер считает физику.
  - Tickrate 30 Hz (настраивается в админке).
  - Client-side prediction + server reconciliation.
  - Lag compensation для выстрелов (откат на RTT/2).

Матчмейкинг:
  - Кнопка «Играть» → очередь по выбранному режиму.
  - Если за 10 секунд оппонента нет → клиенту показывается «Продолжить ожидание / Отмена».
  - В FREE-режиме: опционально (галка в админке) через 30 сек ожидания подсаживается
    бот-оппонент. В CASUAL/STAKE — никогда.
  - MMR существует в БД, но в v1 не используется (ELO=1000 у всех).

Финансы:
  - Все операции через ledger (двойная запись, идемпотентность).
  - При входе в платный матч — ставка резервируется (locked).
  - При завершении: атомарная транзакция (unlock + win + loss + commission).
  - Баланс не может быть < 0.
  - В админке галка: «Можно ли играть с нулевым балансом» (для CASUAL/STAKE).
  - Депозит/вывод через Betra H2H, идемпотентные webhook'и.
  - Вывод: ручной апрув в админке (или авто, если включено в settings).

Прогрессия:
  - 3 стартовых персонажа («Стрелок», «Танк», «Скаут») × по 1 базовому скину.
  - Магазин скинов: цена в монетах (внутренняя валюта) или USD.
  - Loadout: выбор активного персонажа+скина перед матчем.
  - Статистика: сыграно, побед, поражений, заработано, потеряно, MMR.

Античит (server-side):
  - Скорость, ROF, дальность — валидируются на сервере.
  - Input не содержит позиций, только направление.
  - Rate-limit по IP и user_id.
  - AFK-detection: 10 сек без input → технический проигрыш.
  - Реплеи матчей в gzip (input+snapshots), хранятся 30 дней.

UI:
  - PWA, single-page, full-viewport (как мобильная игра, без скролла).
  - Установка на главный экран на iOS/Android.
  - Полноэкранный режим, поддержка safe-area для notch.
  - Single common design system (Tailwind + shadcn/ui).
  - 11 экранов игрока + 9 экранов админки (см. §12).

Деплой:
  - Один git-репозиторий.
  - `docker compose up` — локальная разработка.
  - `docker compose -f compose.prod.yml up -d` — продакшен на чистом VPS.
  - Caddy с авто-SSL (Let's Encrypt).
  - Никаких ручных шагов, только `cp .env.example .env` и редактирование.

Расширяемость (заложить в архитектуру):
  - ECS на game-сервере → легко добавить новые режимы (2v2, 3v3, capture flag).
  - Data-driven контент → новые персонажи/скины без релиза кода.
  - Идемпотентные интеграции → лёгкая замена платёжного провайдера.
  - Reusable workflow в CI → копируется на новые сервисы.

5. АРХИТЕКТУРА
────────────────────────────────────────────────────────────────────────────────

5.1. Поток матча

  [Browser PWA]
       │
       │ HTTPS REST   ──────►  [api: NestJS]  ──────►  [Postgres]
       │ WSS /ws/lobby ─────►  [api: lobby gateway]
       │                          │ (создаёт match, JWT match_token)
       │                          ▼
       │                       [Redis: queue]
       │                          │
       │                          │ matchmaker нашёл пару → match_found
       │                          ▼
       │ WSS /ws/match ────────► [game: uWS]
       │   (match_token)            │
       │                            │ tick 30Hz
       │                            ▼
       │                       (по окончании)
       │                            │
       │                            └─HMAC─► [api /internal/match/finish]
       │                                       │
       │                                       ▼
       │                                  Postgres TX:
       │                                  ledger×4 + match update + stats
       │                                       │
       │                                       └──► lobby WS push: wallet:update

5.2. Caddy роутинг

  /api/*       → api:3000
  /ws/lobby    → api:3000 (websocket upgrade)
  /ws/match    → game:3001 (websocket upgrade)
  /            → web (статика, обслуживается nginx из web-образа)

5.3. Сервисы в compose

  - postgres   (volume: pgdata)
  - redis      (volume: redisdata, appendonly yes)
  - api        (depends_on: postgres, redis; миграции через init)
  - game       (depends_on: api, redis)
  - web        (nginx со статикой, кроме dev — там vite dev server)
  - caddy      (volume: caddy_data, caddy_config)
  - backup     (cron, pg_dump → /backups, опционально)

// (полная схема ниже — реализовать дословно, с миграциями)

enum Role         { PLAYER ADMIN MODERATOR }
enum RoomMode     { FREE CASUAL STAKE }
enum WinCondition { KILL BEST_OF_3 TIMEOUT_HP }
enum MatchStatus  { PENDING RUNNING FINISHED DISPUTED CANCELLED }
enum LedgerType   { DEPOSIT WITHDRAWAL
                    MATCH_STAKE_LOCK MATCH_STAKE_UNLOCK
                    MATCH_WIN MATCH_LOSS COMMISSION
                    ADJUSTMENT BONUS SHOP_PURCHASE }

model User {
  id            Int       @id @default(autoincrement())
  email         String    @unique
  passwordHash  String
  username      String    @unique
  role          Role      @default(PLAYER)
  isBanned      Boolean   @default(false)
  country       String?
  acceptedTosAt DateTime?
  createdAt     DateTime  @default(now())
  meta          Json?

  wallet        Wallet?
  inventory     UserInventory[]
  loadout       UserLoadout?
  ledger        Ledger[]
  stats         UserStats?
  matchesAsP1   Match[]   @relation("p1")
  matchesAsP2   Match[]   @relation("p2")
}

model Wallet {
  userId   Int      @id
  balance  Decimal  @db.Decimal(20,8) @default(0)  // КЕШ
  locked   Decimal  @db.Decimal(20,8) @default(0)
  coins    Int      @default(0)                    // внутренняя валюта
  updatedAt DateTime @updatedAt
  user     User     @relation(fields:[userId], references:[id])
}

model Ledger {
  id              BigInt    @id @default(autoincrement())
  userId          Int
  amount          Decimal   @db.Decimal(20,8)
  type            LedgerType
  refType         String?    // 'match' | 'payment' | 'admin' | 'shop'
  refId           String?
  idempotencyKey  String    @unique
  meta            Json?
  createdAt       DateTime  @default(now())
  user            User      @relation(fields:[userId], references:[id])

  @@index([userId, createdAt])
  @@index([refType, refId])
}

model Room {
  id             Int          @id @default(autoincrement())
  name           String
  mode           RoomMode
  stakeUsd       Decimal?     @db.Decimal(20,8)
  commissionPct  Int          @default(20)
  matchDurationS Int          @default(120)
  winCondition   WinCondition @default(KILL)
  isActive       Boolean      @default(true)
  minBalanceRequired Boolean  @default(true)
  createdAt      DateTime     @default(now())
}

model Match {
  id            String      @id @default(uuid())
  roomId        Int
  player1Id     Int
  player2Id     Int
  player1CharId Int
  player1SkinId Int
  player2CharId Int
  player2SkinId Int
  stakeUsd      Decimal     @db.Decimal(20,8)
  winnerId      Int?
  status        MatchStatus @default(PENDING)
  startedAt     DateTime?
  finishedAt    DateTime?
  replayUrl     String?
  durationMs    Int?
  meta          Json?

  player1       User        @relation("p1", fields:[player1Id], references:[id])
  player2       User        @relation("p2", fields:[player2Id], references:[id])
  room          Room        @relation(fields:[roomId], references:[id])

  @@index([player1Id, finishedAt])
  @@index([player2Id, finishedAt])
  @@index([status])
}

model Character {
  id               Int     @id @default(autoincrement())
  slug             String  @unique
  name             String
  baseHp           Int
  baseSpeed        Float
  baseDamage       Int
  weaponType       String  // 'ranged' | 'rocket' | 'shotgun'
  abilityType      String?
  abilityCooldownS Int     @default(10)
  isActive         Boolean @default(true)
  skins            Skin[]
}

model Skin {
  id            Int      @id @default(autoincrement())
  characterId   Int
  name          String
  rarity        String   // common|rare|epic|legendary
  spriteSetUrl  String
  tint          String?
  statModifiers Json?    // { hpPct?, damagePct?, speedPct? } | null
  priceCoin     Int?
  priceUsd      Decimal? @db.Decimal(20,8)
  isActive      Boolean  @default(true)
  character     Character @relation(fields:[characterId], references:[id])
}

model UserInventory {
  id         Int      @id @default(autoincrement())
  userId     Int
  skinId     Int
  source     String   // purchase|reward|admin|starter
  acquiredAt DateTime @default(now())
  @@unique([userId, skinId])
}

model UserLoadout {
  userId      Int @id
  characterId Int
  skinId      Int
  user        User @relation(fields:[userId], references:[id])
}

model Payment {
  id          String    @id @default(uuid())
  userId      Int
  type        String    // deposit | withdrawal
  status      String    // pending | success | failed | cancelled
  amountUsd   Decimal   @db.Decimal(20,8)
  provider    String    // 'betra-h2h'
  externalId  String?
  meta        Json?
  createdAt   DateTime  @default(now())
  finishedAt  DateTime?
  @@index([userId, createdAt])

  totalEarned    Decimal  @db.Decimal(20,8) @default(0)

7. ФИНАНСОВАЯ ЛОГИКА (КРИТИЧНО)
────────────────────────────────────────────────────────────────────────────────

7.1. Принцип

  - balance в Wallet — это КЕШ.
  - Source of truth: SUM(amount) по ledger для userId.
  - Любая мутация — INSERT в ledger + UPDATE кеша wallet, в одной TX.
  - Каждая запись ledger имеет UNIQUE idempotency_key (защита от двойных проводок).

7.2. Резерв ставки при входе в матч

  TX:
    INSERT INTO ledger (user, amount=-stake, type=MATCH_STAKE_LOCK,
                        refType='match', refId=matchId,
                        idempotencyKey="match:{matchId}:user:{userId}:lock");
    UPDATE wallet SET balance=balance-stake, locked=locked+stake;
    Аналогично для второго игрока.
  COMMIT.

  Если у любого недостаточно средств → всё откатывается, матч не создаётся,
  обоим возвращается queue:retry.

7.3. Завершение матча (НА game-сервере → api /internal/match/finish)

  TX:
    -- Разлок ставок (для обоих)
    INSERT ledger (winner,  +stake, MATCH_STAKE_UNLOCK, key="match:{id}:user:{w}:unlock")
    INSERT ledger (loser,   +stake, MATCH_STAKE_UNLOCK, key="match:{id}:user:{l}:unlock")
    UPDATE wallet[winner], wallet[loser] (locked-=stake, balance+=stake)

    -- Распределение приза
    prize      = stake * 2 * (1 - commission/100)   // например, 1.6$ от банка 2$
    commission = stake * 2 * commission/100         // 0.4$

    INSERT ledger (winner, +prize,      MATCH_WIN,   key="match:{id}:win")
    INSERT ledger (loser,  -stake,      MATCH_LOSS,  key="match:{id}:loss")
    INSERT ledger (system, +commission, COMMISSION,  key="match:{id}:comm")

    UPDATE wallet[winner] balance += (prize - stake)   // чистый приход
    UPDATE wallet[loser]  balance -= 0                  // ставка уже списана при разлоке-возврате
    -- (математика проверяется в тестах)

    UPDATE match SET status=FINISHED, winnerId, finishedAt, durationMs, replayUrl
    UPDATE userStats обоих
  COMMIT.

  Идемпотентность: повторный вызов finish с тем же matchId — no-op
  (по UNIQUE idempotencyKey в ledger).

7.4. Ничья (одновременная смерть, истечение времени с равным HP)

  - Оба ledger MATCH_STAKE_UNLOCK.
  - Без MATCH_WIN/LOSS/COMMISSION (комиссии нет).
  - status=FINISHED, winnerId=null.

7.5. Disconnect / AFK

  - Игрок, не отправлявший input 10 сек или закрывший вкладку — проигравший.
  - Если оба disconnect одновременно → ничья.
  - Если матч не успел стартовать (disconnect до welcome) → CANCELLED + полный
    возврат через MATCH_STAKE_UNLOCK без win/loss.

7.6. Disputed

  - Любая ошибка в TX (race, deadlock, нарушение инварианта) → match.status=DISPUTED.
  - Админ в админке вручную: «вернуть ставки обоим» / «засчитать победу X».
  - Все ручные операции через ledger тип ADJUSTMENT с ref на admin user.

7.7. Депозит (Betra H2H)

  POST /api/wallet/deposit { amountUsd }
    → Payment(pending) → Betra create → возвращает redirect URL
    → клиент идёт оплачивать

  POST /api/payments/betra/callback (webhook, HMAC)
    if status=success:
      TX:
        UPDATE payment SET status=success, finishedAt
        INSERT ledger (DEPOSIT, +amountUsd, key="payment:{paymentId}:deposit")
        UPDATE wallet balance += amountUsd
      COMMIT
      → push wallet:update через lobby WS

  Идемпотентность: повторный webhook с тем же externalId — no-op.

7.8. Вывод

  POST /api/wallet/withdraw { amountUsd, destination }
    → проверки: balance >= amount, KYC если включён, лимиты
    → TX: INSERT ledger (-amountUsd, WITHDRAWAL, status_meta='pending'),
           UPDATE wallet balance -= amount, locked += amount
    → создаём Payment(pending, type=withdrawal)
    → если settings.auto_withdrawal=true → сразу в Betra payout
    → иначе ждёт админ-апрува

  Webhook payout success:
    UPDATE payment success, locked -= amount (списание окончательное)
  Webhook payout failed:
    Отмена: ledger ADJUSTMENT +amount, locked-=amount, balance+=amount.

10. ПРОТОКОЛ МАТЧА (game-WS, бинарный msgpack)
────────────────────────────────────────────────────────────────────────────────

Подключение: wss://<host>/ws/match?token=<matchToken>
matchToken = JWT { matchId, userId, exp }, подписан INTERNAL_SECRET, TTL 30 сек.

Заголовок каждого сообщения: 1 байт типа.

Client → Server:
  0x01 hello       { matchToken }
  0x02 input       { seq, dx:[-1..1], dy:[-1..1], angle:0..2π, fire:bool, ability:bool }
                   Шлётся 30 раз/сек.
  0x03 ping        { t }
  0x04 leave       {}                        (явный выход)

Server → Client:
  0x10 welcome     { matchId, you, opponent, mapW, mapH, tickRate, room }
                   you/opponent: { id, characterId, skinId, username,
                                   stats:{hp,speed,damage,abilityCooldownS},
                                   spawnX, spawnY }
  0x11 snapshot    { tick, ackInputSeq,
                     players:[{id,x,y,angle,hp,ammo,abilityCdMs,buffs?}],
                     bullets:[{id,x,y,owner,type,ttl}],
                     events:[{kind:'shoot'|'hit'|'pickup', ...}] }
  0x12 hit         { victim, attacker, dmg, x, y }   (отдельным сообщением для VFX-точки)
  0x13 match:end   { winnerId|null, reason:'kill'|'timeout'|'disconnect'|'draw',
                     durationMs, score:{...} }
  0x14 pong        { t }
  0x15 error       { code, message }

Снапшоты: full-snapshot 5 Hz, delta-snapshot 30 Hz (delta относительно последнего ack).
Размер delta стремится к < 200 байт.

Реплей:
  game-сервер пишет stream { ts, type, payload } в gzip-файл /tmp/replays/{matchId}.bin.gz
  По окончании — POST в api с replayUrl (если хранилище S3 — uploadStream),
  иначе локально /var/data/replays/, в meta в Match — относительный путь.

────────────────────────────────────────────────────────────────────────────────
11. ИГРОВАЯ ЛОГИКА (game-сервер)
────────────────────────────────────────────────────────────────────────────────

11.1. ECS

  World — массивы компонентов по entityId.
  Components: Position, Velocity, Health, Weapon, Cooldowns, Owner,
              Bullet, Hitbox, AbilityState, AfkTimer, ReplayLog.
  Systems (порядок):
    1. InputApply — применяет последний input игрока к Velocity и желаемому fire/ability
    2. Movement   — Position += Velocity * dt; clamp в карту
    3. Shooting   — если fire+cooldown=0 → spawn Bullet, ROF check
    4. AbilityCast— если ability+cd=0 → активация эффекта
    5. BulletMove — пуля летит, ttl--, проверка границ
    6. Collision  — пули vs игроки (с lag-comp)
    7. Damage     — применение урона, проверка смерти
    8. Cooldown   — уменьшение всех кулдаунов
    9. AfkCheck   — если без input 10 сек — техпоражение
   10. Snapshot   — формирование delta snapshot

11.2. Tick

  setInterval(tickMs, () => world.step(dt));
  tickRate берётся из Settings (default 30Hz).

11.3. Lag compensation

  Каждый игрок хранит ring-buffer позиций последние 1000мс (по тикам).
  При проверке попадания пули, выпущенной игроком A:
    rtt_ms = A.ping
    target_pos = positionsBuffer[B][nowTick - rtt_ms/2/tickMs]
    проверяем коллизию пули с target_pos, не текущей.
  Лимит компенсации: 200мс (любой лаг сверх — игнорируем).

11.4. Server-side валидация input

  - dx, dy ∈ [-1, 1], нормализуем если |v|>1
  - angle ∈ [0, 2π]
  - между двумя input одного игрока должно пройти >= tickMs/2 (защита от спама)
  - input применяется как «желание», все физические значения вычисляет сервер

11.5. Способности (по типу)

  - 'rocket'    — взрывной снаряд, AoE damage, cooldown 10s
  - 'dash'      — рывок 0.3s неуязвимости + ускорение, cooldown 8s
  - 'shield'    — поглощение N урона на 3s, cooldown 12s
  (это базовый набор; новые добавляются как новые системы + Character.abilityType)

11.6. Боты (FREE only)

  В матче с botUser — отдельный AI-controller на сервере, шлёт «виртуальные input»
  каждый тик. Базовая логика:
    - двигаться к цели на расстояние X (между min/max);
    - стрелять в направлении игрока с нормальным шумом;
    - сложность задаётся через Settings ('easy'|'medium'|'hard').

11.7. Реплеи

  Стартуем gzip-stream при welcome.
  Пишем: каждый принятый input + каждый отправленный snapshot.
  При match:end — закрываем, аплоадим/сохраняем, в api передаём replayUrl.

13. БЕЗОПАСНОСТЬ
────────────────────────────────────────────────────────────────────────────────

  - Argon2id для паролей (memoryCost 19MiB, timeCost 2, parallelism 1).
  - JWT access (15 мин) + refresh (30 дней, ротация).
  - Cookies: httpOnly, secure, sameSite=strict, path=/.
  - CSRF: double-submit cookie + X-CSRF-Token заголовок на мутации.
  - Helmet, CORS allowlist (только домен и subdomain admin).
  - Rate-limit (nestjs-throttler):
      auth: 10 req/min/IP
      mutations: 60 req/min/user
      WS-сообщения: 60 msg/sec/socket
  - Idempotency-Key: обязателен на /wallet/* и /shop/buy.
  - HMAC-подпись на /internal/* (SHA-256 от body+timestamp, окно 30 сек).
  - geoip middleware (опционально): блок регистрации из стран, перечисленных
    в Setting.blockedCountries (массив ISO кодов).
  - Логи: pino JSON, без PII. requestId на каждом запросе.
  - Никогда не логируем: passwords, JWT, payment secrets, full ledger amounts с user_id
    (только агрегаты).

[ ] 1.  Регистрация → депозит 1$ через mock Betra → матч за 1$ → корректные

20. ПРАВИЛА КОДА И КОММИТОВ
────────────────────────────────────────────────────────────────────────────────

  - TypeScript strict, noUncheckedIndexedAccess, exactOptionalPropertyTypes.
  - Никаких any (только unknown с narrowing).
  - Все DTO — Zod-схемы в packages/shared.
  - Никаких console.log в production-коде. Только pino logger с ctx.
  - Никаких прямых SQL-запросов кроме сложных отчётов в админке (через Prisma.$queryRaw).
  - Магические числа — только в config/constants. Вся настройка геймплея — в Settings.
  - Файлы < 500 строк. Один класс/сервис — один файл.
  - Conventional commits: feat:, fix:, chore:, docs:, refactor:, test:, perf:.
  - Husky pre-commit: lint, typecheck, format.
  - PR-шаблон: что/зачем/как тестировал.
